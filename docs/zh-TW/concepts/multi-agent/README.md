# 多代理系統

KohakuTerrarium 裡有兩個不同的多代理軸向，它們解決不同的問題。在伸手拿生態瓶之前，先確認你真的需要的是哪一種。

## 縱向 (單體式)

```
      主生物
       /   |   \
    子代理  子代理  子代理
    (規劃)  (實作)  (審查)
```

一隻主生物派遣多個子代理。每個子代理有自己的上下文、自己的提示詞。結果：使用者看到一份對話，父代理背後藏了許多專家對話。

Claude Code、OpenClaw、Oh-My-Opencode 以及大多數現代的 coding agent 都屬於這一類。

- **什麼時候用：** 任務可以很自然地拆成若干階段，而且你想把上下文隔離開。
- **KT 給你的東西：** 子代理是原生功能。在 `subagents[]` 裡設定，用名字呼叫。參見[子代理](../modules/sub-agent.md)。

## 橫向 (模組式)

```
   +-- 生物_a -------+      +-- 生物_b -------+
   |  (某個專家)      | <==> |  (另一個專家)    |
   +-----------------+      +-----------------+
            共用頻道 + 執行期
```

數個獨立的專家生物並排執行，各自有自己的設計。它們透過頻道對話。

CrewAI、AutoGen、MetaGPT 瞄準的就是這個軸。

- **什麼時候用：** 任務符合明確的多角色工作流，而且這些角色真的是不同的 agent (不同提示詞、工具、模型)，不只是一隻 agent 的不同子任務。
- **KT 給你的東西：** [生態瓶](terrarium.md)。生態瓶是托管生物、擁有頻道圖、並在圖變化時處理拓樸 + session 記帳的執行期引擎。它自己不執行 LLM —— 推理活在它內部的生物裡。

## 經驗法則

**先試縱向。** 大部分「我需要多代理」的直覺其實是「我需要上下文隔離」或「我需要一個專家提示詞」，兩者都可以用子代理解決。

當你真的想要**不同的生物**一起合作、且工作流穩定到可以用拓樸表達時，再拿出生態瓶。

## 關於定位

我們把生態瓶定位為橫向多代理的**一種提案架構**。各塊拼圖今天都可以組在一起用：頻道處理選用/條件性的流量，**輸出接線 (output wiring)** 處理確定性的 pipeline 邊 (框架層級的設定，會自動把生物回合結束的文字推進目標的事件佇列 — 不用 `send_message`)，再加上熱插拔、觀察、以及對 root 的生命週期通報。kt-biome 的 `auto_research`、`deep_research`、`swe_team`、`pair_programming` 生態瓶把這些東西完整走過一遍。

我們還在摸索的是**習慣**：什麼時候該用接線、什麼時候該用頻道；條件分支不手刻頻道拼裝怎麼表達；怎麼讓 UI 對接線活動的顯示和頻道流量一樣清楚。這些沒結論的問題放在 [ROADMAP](../../../ROADMAP.md)。

當你真的要讓不同生物合作時用生態瓶。當任務在一隻生物內部可以自然拆解時用子代理 — 對多數「我需要上下文隔離」的直覺而言，縱向比較簡單。框架不替你挑，你自己看情況。
